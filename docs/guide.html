claude-sdlc-setup · operating manual

Claude doesn't improvise your process. It runs one.

Every phase of this pipeline invokes a real methodology — design gets brainstormed and grilled, plans get written task by task, code gets built test-first, and nothing is called done without the command output that proves it. You run six slash commands. The gates do the rest.

Part one

Set it up once

Three steps, about five minutes. You install the /setup skill globally; it scaffolds each repo you point it at.

STEP 01

Install the skill

Run one of these in a terminal. It drops the skill into ~/.claude/skills/setup/ and registers the /setup trigger.

# macOS / Linux
curl -fsSL https://raw.githubusercontent.com/ChanduKaranam/claude-sdlc-setup/main/install.sh | sh

# Windows (PowerShell)
irm https://raw.githubusercontent.com/ChanduKaranam/claude-sdlc-setup/main/install.ps1 | iex
STEP 02

Scaffold a repo

Open any project in Claude Code and type /setup. It detects your stack, runs a short interview (branches, team size, aesthetic, roster), shows you the exact file list, and writes nothing until you approve it. It never overwrites a file that already exists.

/setup
STEP 03

Let the plugins load

The commands run on two plugins — superpowers (brainstorming, TDD, debugging, verification, code review) and ponytail (the YAGNI ladder). The settings.json that /setup writes declares both, so Claude Code offers to install them on your next session in that repo. Your teammates get the same offer when they clone it.

Restart Claude Code, then start your first feature. Until the plugins load, the commands can't call what they depend on.

Part two

What /setup writes

All of it tailored to your stack by the interview. Nothing is generated that has no consumer — no forms skill if nothing renders a form.

CLAUDE.md read every session · the process rules live here .claude/ commands/ the slash commands below agents/ code-reviewer, api-builder, schema-designer… skills/ your conventions + the grilling skill rules/ path-scoped rules, one per surface hooks/ protected-branch guard, session snapshots settings.json permissions, hooks, plugin declarations docs/ tickets/ one file per feature — the design AND the plan STATE.md where the current session left off PLAYBOOK.md for humans, not Claude decisions/ ADRs

The ticket is the artifact. Not a scratch note — it carries the design, the API contracts, the test plan, and an ## Implementation Plan of numbered tasks. /build-ticket executes that plan literally, which is why a vague plan becomes a vague feature, and why a plan containing "TBD" gets rejected outright.

Part three

The pipeline

Each command moves the ticket to a new state and runs the skills listed under it. Pick your track — solo skips the review gates, because there's no second person to hold them.

→ TODOticket created
/new-feature {slug}

Scopes the feature and writes the ticket. It opens by asking whether the feature needs to exist at all — an existing screen, a config change, or a platform behavior that already covers the case is cheaper than any ticket. If it survives that, a short batched interview captures who it's for, which surfaces it touches, and what the empty, loading and error states do.

It scaffolds. It does not design, and it does not build. If a decision can't be made yet, it defers to grooming — not to build time.

Runs ponytail rung 1 — does this need to exist?
TODO → GROOMEDdesign + plan written
/groom-ticket {id}

The deep design pass, and the longest command in the system. Rounds one through four cover the UI walkthrough, the data model, the API contracts and the test scenarios. Round five — risks, dependencies, rollback — is grilled: one question at a time, each with a recommended answer, and facts get looked up rather than asked. It stops when you say you've reached shared understanding, not at a question count.

Then it writes the ## Implementation Plan into the ticket: global constraints copied verbatim, and one task per test cycle with exact file paths, exact signatures, and steps in strict test-first order.

No code here. Not a stub, not a "quick sketch to check the shape".

Runs brainstorming grilling writing-plans
→ IN_PROGRESSdesign + plan written
/work-ticket {slug}

Grooming and claiming, merged. It fills the design sections the scaffold left blank — data model, API shapes, every component state, the test plan — then grills the risks one question at a time, then writes the ## Implementation Plan.

This is the only design gate in the solo track, so it carries the weight that grooming and lead review carry for a team. Nobody else is going to catch a bad design. Don't rush it.

Runs brainstorming grilling writing-plans
GROOMED → APPROVEDor CHANGES_REQUESTED
/review-ticket {id} Leads only

The last gate before code exists — a design flaw caught here costs one conversation; caught after the branch is built it costs the branch. The lead is grilled through the data model, the contracts, the test coverage and the rollback plan. Then a second lens runs over the implementation plan hunting only for what should be cut: an abstraction with one implementation, a dependency added for what five lines do, scaffolding for a future nobody asked for.

Feedback comes back numbered and specific. When the dev re-grooms, they work through it item by item — verifying each against the actual code, pushing back with reasoning where the lead is wrong.

Runs grilling ponytail-review receiving-code-review
APPROVED → IN_PROGRESSclaimed by you
/claim-ticket {id}

Takes an approved ticket off the queue, sets up an isolated workspace, and records the claim so nobody else picks it up. It refuses anything that hasn't cleared lead review — the approval gate has no bypass, including for the person who wrote the ticket.

Runs using-git-worktrees
IN_PROGRESScode gets written
/build-ticket {slug} This is where code happens

Everything before this was design. This command takes an isolated workspace, runs the suite once to prove the baseline is green, then walks the ticket's plan task by task:

RED    write the failing test — run it — confirm it fails for the right reason
GREEN  minimal code to pass — run it — confirm the output is pristine
       refactor · commit · review the diff · next task

You'll be offered two modes. Subagent-driven spins up a fresh agent per task and reviews each one as it lands — the default, and usually right. Inline keeps everything in your session if you want to watch closely.

If something breaks it finds the root cause instead of stacking fixes. If three fixes have failed, it stops and tells you the design is wrong — it does not attempt a fourth.

Runs subagent-driven-development test-driven-development systematic-debugging requesting-code-review ponytail
→ READY_FOR_REVIEWPR opened
/complete-feature {slug}

The pre-merge gate. Every checkbox here demands the command output that proves it, in the message that claims it — "it passed in build" doesn't count, and neither does "should be fine". Tests, lint, types, the test-plan files actually existing on disk.

Two reviews run on the diff. One hunts correctness bugs. The other hunts only for what should be deleted. Then every deliberate shortcut you left behind gets harvested into the PR body with its ceiling and its upgrade trigger, so the corners you cut are visible to whoever merges.

A failing test is a debugging job, not a .skip job. The PR does not open while anything is red.

Runs verification-before-completion requesting-code-review ponytail-review ponytail-debt finishing-a-development-branch
MERGED → DEPLOYEDpromotion
/dev-deploy · /prod-deploy

Promote the integration branch to dev, then dev to production with a version tag. Production has a hard CI gate: red on the source branch and the PR doesn't open.

After either merge, the smoke test is not a checkbox. Hit the deployed thing — health, one real read, one real write — and read the responses. A green pipeline proves the pipeline ran, not that the app serves traffic.

Runs verification-before-completion

/handoff

Pauses mid-build. It runs the tests before writing down what's done, so the state file is true rather than optimistic. A half-finished file is marked in progress, not done. A red suite goes in the blockers list, not left out because you meant to fix it next session.

/resume-session

Restores context from where you left off. If there were blockers, it debugs them first — a blocker is the reason the last session stopped, not a note to route around. On GO it hands straight back to /build-ticket, which picks the plan up mid-flight.

Part four

When something shipped is broken

Bugs don't re-enter at the top of the pipeline — a two-line fix shouldn't drag a UI walkthrough behind it. /fix-bug is the loop back into shipped code, and it holds the same line the build phase does.

/fix-bug {slug}

A bug report names a symptom. It does not name a cause, and it does not name a fix — even when it sounds like it does, and even when you wrote it.

01

It grills the report

One question at a time: what you expected, what happened instead, the exact steps, every time or only sometimes, the error text verbatim. It looks up whatever it can find for itself — the parent ticket, when the code last worked, whether any test covers that path — and only asks you the things it genuinely cannot know.

02

It reproduces it before theorising

Consistently, on the fix branch, before a single hypothesis. If it cannot reproduce the bug, it stops and tells you — it does not ship a plausible fix for something nobody has observed. A fix for a bug you never saw is a guess wearing a commit message.

03

It traces the root cause — then stops

It works backward from the symptom to the first place the data is actually wrong. Everything downstream of that point is symptom, including the line the error was thrown on. One hypothesis at a time; fixes are never stacked.

Then it stops and hands you a diagnosis, with the evidence that proves it and — the part that matters — the blast radius:

BUG-014 — DIAGNOSIS

Root cause:   src/lib/parseRange.ts:41 — parseRange()
Why:          empty filter yields [NaN, NaN], not [0, 0]. Traced, not guessed.
Evidence:     {the instrumented output}

Blast radius — every caller of parseRange():
  routes/search.ts:88    also broken · covered by this fix
  routes/export.ts:23    also broken · NOT in the report
  jobs/digest.ts:12      not broken · guards upstream

  Patching only the reported path would leave /export broken.

Confirm this diagnosis before I write the fix.

Nothing gets written until you confirm it. That's the whole point of the step: a bug report names one broken path, and the root cause usually breaks several. The callers nobody reported are the ones that bite in three weeks.

04

The failing test comes before the fix

It must reproduce the reported symptom, fail against the unfixed code, and fail for the right reason — not because of a typo in the test. If it passes against the unfixed code, the bug was never reproduced, and it goes back to step 02.

05

It fixes the cause, once

At the root, where the callers route through — not once per caller. The lazy fix and the correct fix are the same fix here: one guard in the shared function is a smaller diff and the one that repairs the siblings. It won't refactor while it's in there, won't add a dependency, and won't harden adjacent code you didn't ask about.

06

It proves the red-green

Reverts the fix, watches the test go red, restores it, watches it go green — and pastes both. A regression test you have never watched fail is not testing anything; it will quietly stop covering the bug the first time someone refactors past it.

Then /complete-feature ships it through the same gates a feature gets. A one-line fix that skips the gates is exactly how the next bug gets in.

⨯ BLOCKED
No fix before the diagnosis is confirmed

Not a test, not a stub, not "just trying something". There are no small bugs that earn an exemption — "it's a one-liner" is the sentence people say right before they patch the wrong line. And three failed fixes means stop, not fix number four: at three, the design is wrong, and more poking at the code will not reveal that.

Your first feature, end to end

What you actually type, and what happens between the lines. Solo track shown; a team inserts grooming, lead review and a claim before the build.

       ✗ FAIL  test written first, and it fails

     coming back?   /resume-session

── ship ────────────────────────────────────────────────
/complete-feature saved-searches
   ↳ every gate produces its own evidence.
     two reviews on the diff. shortcuts surfaced. PR opened.

── later, something's broken ───────────────────────────
/fix-bug saved-searches-empty-filter
   ↳ grills the report. reproduces it. traces the cause.
     then STOPS — you confirm the diagnosis and its blast
     radius before a line is written.
     then: failing test → fix at the root → prove red-green.
     then /complete-feature, same gates as a feature.

The one thing people get wrong: treating the plan as a formality and moving on. /build-ticket executes those tasks literally — the file paths, the signatures, the order. Fifteen minutes spent reading the plan when grooming ends is the highest-leverage time in the whole cycle. If the plan is wrong, go back and fix the plan. Do not fix it inline and build from your own version.

Part six

The four things it won't let you do

These aren't style guidance. They're the gates — the commands stop rather than proceed past them.

⨯ BLOCKED
No production code without a failing test first

Write the test, run it, watch it fail for the right reason — a test that fails on a typo is not a red test. Only then write the code. If code got written before its test, it gets deleted. Not kept as reference, not adapted.

/build-ticket · test-driven-development
⨯ BLOCKED
No fixes without root cause investigation first

Reproduce it before you theorize. One hypothesis, one variable, one smallest-possible change — never stack fixes on top of each other. And after three failed fixes it stops entirely: the design is wrong, not the code, and fix number four won't find that out.

/build-ticket · /resume-session · systematic-debugging
⨯ BLOCKED
No completion claims without fresh verification evidence

If the command wasn't run in the message making the claim, the claim can't be made. Not "I ran it earlier". Not "it should pass". A checkbox ticked without running its command is a lie in a file that outlives you.

/complete-feature · /handoff · the deploys
⨯ BLOCKED
No code during design

Grooming designs. It does not stub, sketch, or scaffold "just to check the shape". And no plan ships with a TBD in it — /build-ticket refuses the ticket, because an unresolved decision does not get more resolved by having code written around it.

/groom-ticket · /work-ticket · brainstorming
Part seven

The ladder, before every diff

Ponytail runs across the whole pipeline. Before writing anything, it climbs — and stops at the first rung that holds.

01
Does this need to exist at all? Speculative work gets skipped, and said so in one line.
02
Is it already in this codebase? Grep before you write. Re-implementing what lives two files over is the most common failure.
03
Does the standard library do it?
04
Does a native platform feature cover it? A date input over a picker library. A database constraint over application code.
05
Does something already installed solve it? Never add a dependency for what a few lines do.
06
Can it be one line? Then it's one line.
07
Only then: the minimum code that works.

Shortcuts are tracked, not hidden

When a corner is deliberately cut, it leaves a marker naming the ceiling and what would trigger the upgrade. /complete-feature harvests every one of them into the PR body, so the reviewer sees the cost, not just the code.

// ponytail: in-memory cache, move to
// Redis if this goes multi-instance

Where it is never lazy

  • Input validation at trust boundaries
  • Error handling that prevents data loss
  • Security
  • Accessibility basics
  • Anything you explicitly asked for

And never lazy about understanding the problem. The ladder shortens the solution, never the reading.
